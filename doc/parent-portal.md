# Parent portal

[← Wiki home](../README.md)

## Audience

Logged-in **parents and guardians** (users with the **Parent** role, and optionally **Spouse** users on a family account). This is the primary portal for families after [registration](registration-user-fields.md).

Parents who also hold **Teacher**, **Staff**, or other roles use the same login; the portal exposes the right tools per role without a second account (see [RBAC](rbac.md)). Mobile-first UX is expected because **phone + SMS** is a primary login path.

## Purpose

The parent portal is the **family command center**: manage the account, students, enrollment, payments, and visibility into each child’s school activity—without requiring the [student portal](student-portal.md) unless the child has their own login or the school enables assisted submission.

---

## Backend configuration

Dynamic portal data (courses, cart, announcements, schedules) is configured in admin — see **[Frontend ↔ backend configuration map](frontend-backend-config.md#parent-portal)**.

| Surface | Admin / teacher source | Detailed behavior |
|---------|------------------------|-------------------|
| Catalog & pricing | Academic year, pricing rules | Cart recalculates when discounts apply. |
| Enrollment state | Rosters + payment records | Dashboard badges for unpaid seats. |
| Child schedule | Master schedule + exceptions | Per-child tab; multi-child families see switcher. |
| Class visibility | Teacher course publish | Mirrors student-visible subset plus parent-safe fields. |

## Primary features

### Family dashboard

- Overview of all **students** on the account with enrollment status chips
- **Family Identifier** and account summary for office reference
- Quick links: enroll in classes, pay balance, view announcements
- School-wide **announcements** and activity feed (read)
- Alerts: unpaid balance, upcoming registration deadline, missing profile fields

**Edge cases**

| Situation | Behavior |
|-----------|----------|
| Two children, one unpaid | Dashboard highlights unpaid student; checkout may still be per-cart rules. |
| Spouse login | Sees same children; payment button disabled if not primary owner. |
| New student mid-year | Add child profile then enroll; prior year history separate. |

### Account & profile management

| Capability | Primary owner | Other parent (Spouse) | Edge cases |
|------------|---------------|------------------------|------------|
| Edit own profile (nickname, contact, WeChat, email, address) | Yes | Yes | WeChat used for future notifications only until integrated. |
| Add / remove **Spouse** users on account | Yes | TBD (future: limited access) | Cannot remove primary without transfer flow. |
| Add / edit / remove **student** profiles | Yes | Yes (configurable) | Removing student with active enrollment blocked until drop policy met. |
| Manage linked [login methods](authentication.md) | Yes | Own login only | Spouse manages own OAuth/phone links. |
| View **Family Identifier** | Yes | Yes | Display-only; generated by system. |

Profile fields: [Registration — user fields](registration-user-fields.md).

### Student management

For each child on the account:

- View and edit student profile (names EN/CN, gender, DOB, regular school, grade)
- View **enrolled classes** for the current year with teacher and schedule summary
- Open **read-only** (or assisted) view of student schedule, assignments, and progress — see [Student visibility](#student-visibility)
- Start **new enrollment** for upcoming term from catalog

**Workflow — add a child and enroll**

1. Primary or permitted parent opens **Students → Add child**.
2. Parent completes required profile fields and saves.
3. Parent opens **Enroll** for that child; catalog filters by placement hints (DOB, grade).
4. Parent adds sections to cart; discounts apply at cart review.
5. Primary owner completes checkout; confirmation email/receipt stored.

### Registration & enrollment (phase 1)

Central to phase 1 delivery ([Registration & payment](registration-payment.md)):

1. Browse **course catalog** by grade / level (using student DOB and regular-school grade as hints)
2. Select classes per student → **cart** (multi-student cart supported)
3. Apply **discounts** (early bird, sibling, multi-class) with transparent line items
4. **Checkout and pay** (primary owner only) via configured gateway
5. View confirmation, receipts, and enrollment status; offline payment shows when admin marks paid

Non-primary parents may browse and build a cart; **payment** requires primary owner unless school changes policy.

| Edge case | Behavior |
|-----------|----------|
| Class full at checkout | Block with waitlist message if enabled. |
| Price change after cart built | Prompt to refresh cart before pay. |
| Partial payment | School policy: block enrollment or allow hold state. |

### Payments & billing

| Feature | Notes | Acceptance criteria |
|---------|--------|---------------------|
| Payment history | Per student and per term | Export or resend receipt for primary owner. |
| Receipts | Download or resend | Matches gateway transaction id when online. |
| Unpaid / balance due | Alerts on dashboard | Clears when admin marks offline paid or gateway succeeds. |
| Offline payments | Admin-marked paid — visible to parent | Shows paid date and method note without card data. |

### Student visibility

Parents should see what matters for each child without full teacher tools:

| View | Content | Detailed behavior | Edge cases |
|------|---------|-------------------|------------|
| **Schedule** | Daily/weekly class times, teacher, room | Same exception data as student portal. | Cancelled class labeled clearly. |
| **Assignments** | Due dates, submission status | Exact scores TBD per school policy. | Parent cannot grade; optional proxy upload TBD. |
| **Progress** | Grades and completion summary where teachers publish to families | Hidden until teacher returns grade if policy set. | Sibling data never mixed on same screen without switcher. |
| **Class announcements** | Posts from teacher/TA for enrolled classes | Union across all children. | Dropped class stops new posts; history may remain read-only. |

*Young students:* if the child has no login, the parent portal is their main window into LMS activity. If the child has a student login, parents still need this view for oversight.

### Announcements & communication

- **School-wide** feed (from admin/staff) on dashboard
- **Class-level** items for each enrolled student, tagged by child when multiple
- Optional: WeChat ID on file for future notification channel (field collected at registration)
- Email notifications for critical posts when enabled in settings

### Settings

- Notification preferences (email; SMS/WeChat when integrated)
- Language preference (English / Chinese UI — recommended)
- Switch active **role context** when user is also Teacher or Volunteer
- Session security: linked login methods, logout all devices optional

## Combined roles (same login)

| User is also… | Portal behavior | Acceptance criteria |
|---------------|-----------------|---------------------|
| **Teacher** | Link or menu to [Teacher portal](teacher-portal.md) | One sign-in reaches teacher workspace. |
| **Volunteer / Staff** | Duty schedule, announcement tools per permissions | Volunteer without post permission sees read-only feed. |
| **Student** (uncommon for parent account) | Link to [Student portal](student-portal.md) for own classes | Student data isolated from parent children list. |
| **Admin** | Link to [Admin portal](admin-portal.md) if applicable | Admin menus not shown without admin role. |

Use one session; do not require a second account.

## Requirements

| ID | Requirement | Detailed behavior | Edge cases | Acceptance criteria | Status |
|----|-------------|-------------------|------------|---------------------|--------|
| REQ-PAR-01 | Dedicated **parent portal** for family account management. | Separate nav from student/teacher; defaults to family dashboard. | Combined roles show role switcher. | Parent lands on dashboard after login. | Confirmed |
| REQ-PAR-02 | Parents manage **student profiles** and view all children on one dashboard. | CRUD per permissions; spouse rules apply. | Cannot attach child to two accounts. | Two children listed with correct enrollments. | Confirmed |
| REQ-PAR-03 | **Primary owner** completes payments; portal shows billing to all parents on account. | Pay button gated; history visible to spouse. | Failed card leaves enrollment unpaid state visible. | Spouse sees receipt; cannot pay unless policy changed. | Confirmed |
| REQ-PAR-04 | Parents complete **course enrollment** and cart checkout from the portal. | Multi-student cart; discounts at review. | Full class blocked at payment. | Successful pay creates enrollment rows. | Confirmed |
| REQ-PAR-05 | Parents view each child’s **schedule, assignments, progress, and class announcements**. | Per-child tabs; read-only. | Grade detail level follows school TBD flag. | Schedule matches student view for same child. | Confirmed |
| REQ-PAR-06 | Parents see **school-wide** announcements and feeds. | Filtered server-side. | Non-enrolled parent still sees school-wide when logged in. | School closure post visible on dashboard. | Confirmed |
| REQ-PAR-07 | Users with Parent + other roles access other portals without re-login. | Shared session cookie / token. | Role switch clears stale UI state. | Parent+teacher reaches both portals in one session. | Confirmed |
| REQ-PAR-08 | Profile and family fields match [user field spec](registration-user-fields.md). | Validation on required fields; bilingual names stored. | Missing DOB may block placement hints with warning. | Saved profile round-trips all required fields. | Confirmed |
